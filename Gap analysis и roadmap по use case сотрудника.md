# Gap analysis и roadmap по use case сотрудника

## Назначение документа

Документ связывает целевой сценарий из `Use Case сотрудника и чек-лист соответствия.md` с текущим состоянием проекта.

Цель документа:

- зафиксировать, что уже реализовано
- отделить частично реализованные зоны от отсутствующих
- задать порядок доработок
- использовать roadmap как рабочий backlog для следующих итераций

## Базовый источник требований

Основной эталон сценария:

- `Use Case сотрудника и чек-лист соответствия.md`

Дополнительные источники:

- `Вопросы и ответы по проекту.md`
- `План реализации и архитектура проекта.md`
- `ТЕХНИЧЕСКОЕ ЗАДАНИЕ.pdf`

## Текущее покрытие use case

### 1. Вход сотрудника в систему

Статус:

- реализовано

Что уже есть:

- вход по логину и паролю
- JWT-аутентификация
- роль `сотрудник`

### 2. Загрузка скана, фото или PDF заказ-наряда

Статус:

- реализовано частично

Что уже есть:

- загрузка PDF и изображений
- сохранение файла в хранилище
- создание документа и черновика ремонта

Что не хватает:

- жёсткой валидации содержимого файла
- ограничений на размер и защитных лимитов
- явного сценария multi-photo в единый документ

### 3. OCR и извлечение данных

Статус:

- реализовано частично

Что уже есть:

- извлечение текста из PDF
- OCR для изображений и сканов
- извлечение ключевых полей
- разбор строк работ и запчастей

Что не хватает:

- платформенно-независимого OCR контура
- фоновой очереди обработки
- детерминированной и безопасной транзакционной модели при ошибках OCR

### 4. Экран ручной проверки сотрудником

Статус:

- реализовано частично

Что уже есть:

- просмотр документа
- просмотр распознанных полей
- ручное изменение части полей ремонта
- подтверждение сотрудником

Что не хватает:

- формализованного единого workflow именно под сценарий сотрудника
- полного UX для последовательной проверки всех типов несоответствий
- более явного разделения этапов: распознавание, корректировка, подтверждение, итоговый отчёт

### 5. Разделение на работы и запчасти

Статус:

- реализовано частично

Что уже есть:

- извлечение строк работ
- извлечение строк запчастей
- сохранение в отдельные сущности

Что не хватает:

- гарантированного качества разбиения на всех шаблонах документов
- отдельной проверки полноты разбиения по документу

### 6. Сверка машины с базой техники

Статус:

- реализовано частично

Что уже есть:

- поиск и привязка техники
- placeholder-механизм для непривязанных документов
- ручная перепривязка техники

Что не хватает:

- явного бизнес-предупреждения сотруднику в терминах use case
- отдельного статуса или блока отчёта “машина не найдена”

### 7. Сверка сервиса с базой сервисов

Статус:

- реализовано частично

Что уже есть:

- справочник сервисов
- предварительные и подтверждённые сервисы
- ручное назначение сервиса

Что не хватает:

- явного сценария предупреждения “сервис отсутствует в базе”
- более чистого разделения между справочником и автосинхронизацией

### 8. Проверка по нормо-часам

Статус:

- реализовано частично

Что уже есть:

- справочник нормо-часов
- сопоставление работ со справочником
- проверки и сигналы по отклонениям

Что не хватает:

- полного покрытия use case на уровне финального отчёта сотрудника
- прозрачного UX, где сотрудник видит причину несопоставления по каждой строке

### 9. Сравнение с историей по аналогичным работам, сервисам и машинам

Статус:

- реализовано частично

Что уже есть:

- отдельные проверки на повторяемость и аномалии
- накопление исторических ремонтов
- дашборды качества данных

Что не хватает:

- явного модуля сравнительного анализа “эта работа против истории”
- консолидированной логики сравнения по сервисам и по типам техники
- понятной интерпретации отклонения в итоговом отчёте

### 10. Проверка сумм и других параметров ТЗ

Статус:

- реализовано частично

Что уже есть:

- проверки итогов
- проверки строк
- checks в ремонте

Что не хватает:

- единой матрицы всех обязательных проверок из ТЗ
- явной группировки предупреждений по типам несоответствий

### 11. Итоговый отчёт сотрудника по конкретному заказ-наряду

Статус:

- не реализовано в целевом виде

Что уже есть:

- детализация ремонта
- история изменений
- экспорт ремонта

Что не хватает:

- единого итогового отчёта по заказ-наряду
- отдельного блока предупреждений и несоответствий
- финальной сводки “что соответствует / что не соответствует / что требует ручного решения”

## Главные разрывы между целевым use case и текущей реализацией

Ключевые разрывы:

- нет единого финального отчёта сотрудника по заказ-наряду
- OCR-пайплайн зависим от Apple-специфичных инструментов
- обработка документа выполняется прямо в HTTP-запросе
- предупреждения по машине, сервису, нормо-часам и истории не собраны в единый бизнес-слой
- нет явной pre-deploy проверки, автоматизированной на уровне тестов и acceptance сценариев

## Roadmap

### P0. Стабилизация критического потока заказ-наряда

Цель:

- сделать сценарий сотрудника предсказуемым и безопасным для продовой эксплуатации

Задачи:

- перевести OCR и обработку документа в фоновый асинхронный контур
- убрать зависимость критического OCR от Apple-only инструментов
- ввести ограничения на размер, тип и сигнатуру загружаемых файлов
- выровнять транзакции при ошибках обработки документа
- сделать healthcheck, который реально проверяет БД и storage

Критерий готовности:

- загрузка документа не зависит от длительности OCR
- ошибка OCR не ломает состояние ремонта
- документ остаётся в контролируемом workflow даже при сбое распознавания

### P1. Доведение сценария ручной проверки сотрудника

Цель:

- собрать последовательный workflow сотрудника без логических разрывов

Задачи:

- выделить единый экран или последовательность экранов проверки заказ-наряда
- сделать явные предупреждения:
  - машина не найдена
  - сервис не найден
  - нормо-часы не сопоставлены
  - суммы или структура документа вызывают сомнение
- отделить технические статусы от бизнес-предупреждений
- привести права сотрудника к согласованной бизнес-модели

Критерий готовности:

- сотрудник проходит сценарий от загрузки до подтверждения без перехода в “технические” сущности и без неоднозначных статусов

### P2. Итоговый отчёт по заказ-наряду

Цель:

- дать сотруднику и администратору финальный результат обработки документа в одном представлении

Задачи:

- собрать консолидированный отчёт по заказ-наряду
- включить в отчёт:
  - реквизиты
  - машину
  - сервис
  - работы
  - запчасти
  - суммы
  - предупреждения
  - несоответствия по нормо-часам
  - аномалии по истории
- добавить экспорт отчёта в человекочитаемом формате

Критерий готовности:

- по любому заказ-наряду сотрудник получает единый итоговый результат с явным перечнем отклонений

### P3. Аналитика и историческое сравнение

Цель:

- превратить накопленные данные в полезный слой контроля ремонтов

Задачи:

- нормализовать базу сравнения работ и запчастей
- реализовать сравнение однотипных работ:
  - по сервисам
  - по машинам
  - по периодам
- показывать историю отклонений не только в dashboard, но и в карточке конкретного заказ-наряда

Критерий готовности:

- аномалии по работам и стоимости видны и на аналитическом уровне, и в контексте конкретного документа

### P4. Контур качества и pre-deploy соответствие

Цель:

- сделать соответствие use case проверяемым, а не только декларативным

Задачи:

- добавить acceptance checklist в процесс релиза
- добавить backend smoke tests по критическому сценарию сотрудника
- добавить integration tests на:
  - загрузку документа
  - OCR failure
  - ручную корректировку
  - сверку машины и сервиса
  - отчёт по заказ-наряду
- добавить manual regression checklist перед деплоем

Критерий готовности:

- перед деплоем команда может явно ответить, не сломан ли базовый сценарий сотрудника

## Рекомендуемый порядок работ

1. `P0` стабилизация runtime и обработки документа
2. `P1` доведение рабочего сценария сотрудника
3. `P2` реализация итогового отчёта по заказ-наряду
4. `P4` автоматизация проверки соответствия
5. `P3` углубление аналитики и исторического сравнения

## Правило сопровождения документа

- если меняется целевой сценарий сотрудника, сначала обновляется `Use Case сотрудника и чек-лист соответствия.md`
- если меняется приоритет или объём работ, обновляется этот roadmap
- после каждой значимой правки перед деплоем нужно проверять:
  - соответствует ли изменение use case
  - закрывает ли оно один из разрывов roadmap
  - не сломало ли оно ранее закрытый пункт
