# Use Case сотрудника и чек-лист соответствия

## Назначение документа

Документ фиксирует целевой рабочий сценарий сотрудника для MVP и служит обязательным чек-листом соответствия перед деплоем.

Это не справочный текст, а рабочий acceptance baseline для всех изменений, влияющих на:

- загрузку и обработку заказ-наряда
- ручную проверку сотрудником
- валидацию машины, сервиса, работ, запчастей и сумм
- итоговый отчёт по конкретному заказ-наряду

## Основной Use Case

### Название

Сотрудник загружает скан заказ-наряда, проверяет результат распознавания, вносит корректировки и получает итоговый отчёт по заказ-наряду с предупреждениями о несоответствиях.

### Актор

- сотрудник

### Предусловия

- сотрудник имеет активную учётную запись
- сотрудник вошёл в систему
- сотруднику доступна техника в зоне его ответственности либо доступна загрузка нового документа в черновик
- в системе доступны:
  - база техники
  - база сервисов
  - справочник нормо-часов
  - накопленная база исторических ремонтов, работ, запчастей и сумм

### Триггер

Сотрудник начинает обработку нового заказ-наряда и загружает скан, фото или PDF документа.

### Основной сценарий

1. Сотрудник входит в систему.
2. Сотрудник открывает экран загрузки заказ-наряда.
3. Сотрудник загружает скан, фото или PDF документа.
4. Система сохраняет исходный файл и запускает OCR и извлечение данных.
5. Система выделяет из документа:
   - реквизиты заказ-наряда
   - дату ремонта
   - машину
   - сервис
   - пробег
   - список работ
   - список запчастей
   - суммы по строкам и итоговые суммы
6. Система создаёт черновик ремонта и показывает сотруднику результат распознавания.
7. Сотрудник просматривает исходный документ и распознанные поля.
8. Сотрудник проверяет корректность распознанных данных.
9. Сотрудник при необходимости вручную исправляет или дополняет:
   - машину
   - сервис
   - дату
   - номер заказ-наряда
   - пробег
   - работы
   - запчасти
   - суммы
10. Система разделяет содержимое документа на:
    - ремонтные работы
    - запчасти
11. Система сверяет машину с базой техники.
12. Система сверяет сервис с базой сервисов.
13. Если машина отсутствует в базе, система явно формирует предупреждение о несоответствии.
14. Если сервис отсутствует в базе, система явно формирует предупреждение о несоответствии.
15. Система сопоставляет работы со справочником нормо-часов.
16. При расхождении по нормо-часам система формирует предупреждение о несоответствии.
17. Система проверяет суммы, структуру строк, дубли, повторяемость и другие контрольные параметры из технического задания.
18. Система сравнивает стоимость и состав аналогичных работ:
    - в разных сервисах
    - на разных машинах
    - в исторических ремонтах
19. Система накапливает результаты проверки в базе как историю для последующего сравнения и аналитики.
20. Сотрудник получает итоговый отчёт по конкретному заказ-наряду.
21. В отчёте отображаются все параметры, не соответствующие норме, с явным предупреждением.
22. Сотрудник завершает обработку документа и передаёт результат дальше по workflow подтверждения.

### Альтернативные сценарии

#### A1. Документ распознан частично

- система помечает документ как требующий проверки
- сотрудник вручную дополняет недостающие поля
- документ остаётся в рабочем процессе до заполнения обязательных данных

#### A2. Машина не найдена в базе

- система не подтверждает документ автоматически
- система формирует явное предупреждение
- сотрудник выбирает существующую машину или передаёт случай на дополнительную проверку

#### A3. Сервис не найден в базе

- система формирует явное предупреждение
- сотрудник выбирает существующий сервис либо создаёт предварительный сервис по утверждённым правилам
- случай остаётся видимым как отклонение до подтверждения

#### A4. Нормо-часы не сопоставились

- система формирует предупреждение по конкретной работе
- сотрудник видит, какая позиция не прошла сопоставление
- случай остаётся в отчёте как отклонение до ручного решения

#### A5. Найдены ценовые или структурные аномалии

- система показывает отклонения по суммам, составу, повторяемости и иным контрольным параметрам
- документ не считается полностью чистым до ручной проверки

#### A6. OCR завершился ошибкой

- система сохраняет исходный файл
- документ переводится в состояние ручной проверки
- сотрудник видит, что автоматическое распознавание не дало надёжного результата

### Обязательные результаты сценария

По завершении сценария система должна иметь:

- сохранённый исходный документ
- созданный или обновлённый черновик ремонта
- список работ
- список запчастей
- результат сверки машины
- результат сверки сервиса
- результат проверки нормо-часов
- результат проверок сумм и аномалий
- итоговый отчёт по конкретному заказ-наряду
- журнал предупреждений и несоответствий

## Требования к итоговому отчёту по заказ-наряду

Итоговый отчёт по конкретному заказ-наряду должен включать:

- идентификатор и номер заказ-наряда
- дату ремонта
- машину
- сервис
- пробег
- перечень работ
- перечень запчастей
- итоговые суммы
- предупреждения по отсутствующей машине
- предупреждения по отсутствующему сервису
- предупреждения по несоответствию нормо-часам
- предупреждения по суммам и иным параметрам контроля
- итоговый статус проверки

## Чек-лист соответствия перед деплоем

Перед каждым деплоем все изменения, затрагивающие загрузку документов, OCR, проверку ремонтов, справочники, аналитику и отчёты, должны быть проверены на соответствие этому use case.

### Блок 1. Входной сценарий сотрудника

- сотрудник может войти в систему
- сотрудник может загрузить скан, фото или PDF заказ-наряда
- исходный файл сохраняется без потери
- после загрузки запускается автоматическая обработка документа

### Блок 2. OCR и извлечение данных

- система извлекает ключевые поля заказ-наряда
- система извлекает работы и запчасти отдельно
- при частичном распознавании документ не теряется и остаётся доступным для ручной проверки
- при ошибке OCR документ переводится в контролируемый статус, а не пропадает из workflow

### Блок 3. Ручная проверка сотрудником

- сотрудник может проверить распознанные данные
- сотрудник может исправить обязательные поля
- сотрудник может исправить работы, запчасти и суммы
- изменения сотрудника сохраняются в системе

### Блок 4. Проверка справочников

- система сверяет машину с базой техники
- система сверяет сервис с базой сервисов
- отсутствие машины формирует явное предупреждение
- отсутствие сервиса формирует явное предупреждение

### Блок 5. Нормо-часы

- система сопоставляет работы со справочником нормо-часов
- несоответствия по нормо-часам отображаются явно
- сотрудник видит, какая именно работа вызвала предупреждение

### Блок 6. Проверка сумм и аномалий

- система проверяет итоговые суммы и суммы по строкам
- система сравнивает аналогичные работы по историческим данным
- система показывает отклонения по стоимости, структуре и повторяемости
- результаты проверки сохраняются в отчёте по заказ-наряду

### Блок 7. Итоговый результат

- по каждому заказ-наряду формируется единый итоговый отчёт
- отчёт содержит список всех несоответствий и предупреждений
- сотрудник видит итоговый статус проверки документа
- данные пригодны для следующего этапа подтверждения и аналитики

## Правило использования документа

- любые изменения по документам, OCR, review-flow, справочникам, проверкам и отчётности должны сверяться с этим документом
- перед деплоем нужно явно пройтись по чек-листу и зафиксировать, что use case не нарушен
- если новая реализация меняет сценарий сотрудника, сначала нужно обновить этот документ, затем код, затем проверку перед деплоем
