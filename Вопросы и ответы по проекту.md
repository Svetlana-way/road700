## Журнал уточнений

### Вопрос 1

Какой технологический стек утверждаем для MVP: frontend, backend, база данных, хранение файлов и способ деплоя?

### Ответ 1

Стек выбирает исполнитель. Принято решение делать MVP на следующем наборе технологий:

- Frontend: React + TypeScript + Vite
- UI: React Admin / MUI как основа для быстрого построения внутренних интерфейсов
- Backend: Python + FastAPI
- ORM и миграции: SQLAlchemy + Alembic
- База данных: PostgreSQL
- Аутентификация: JWT + bcrypt
- Хранение файлов: S3-совместимое хранилище, для локальной разработки допускается файловая система
- Деплой: Docker Compose на Linux-сервере

Причина выбора: этот стек оптимален для быстрого запуска внутренней системы с большим количеством таблиц, фильтров, форм, аналитики и понятной серверной логикой.

Нужно ли в MVP делать только ручное добавление ремонта, или уже в первой версии требуется автоматическое извлечение данных из PDF заказ-наряда?

### Ответ 2

В MVP обязательно требуется автоматическое извлечение данных из PDF заказ-наряда.

Следствие для реализации:

- модуль загрузки файла должен запускать парсинг PDF
- система должна уметь предзаполнять поля ремонта данными из документа
- потребуется отдельный контур валидации распознанных данных и ручной корректировки пользователем

- потребуется механизм ручной проверки и исправления распознанных данных
- желательно накапливать шаблоны и правила разбора по конкретным сервисам

### Вопрос 7

После распознавания PDF система должна автоматически создавать ремонт сразу в базе или сначала только заполнять черновик для проверки пользователем?

### Ответ 7

Для MVP лучше сначала создавать черновик ремонта для проверки пользователем, а не сразу финальную запись в базе.

Причины:

- при OCR на сканах и фотографиях неизбежны ошибки распознавания
- документы приходят от разных сервисов в разных форматах
- черновик снижает риск появления неверных ремонтов, сумм, работ и материалов
- пользователь сможет быстро подтвердить, исправить или дополнить распознанные поля

Рекомендуемый сценарий:

- пользователь загружает PDF
- система распознаёт документ
- создаётся черновик ремонта с заполненными полями и пометками уверенности
- пользователь проверяет и подтверждает запись
- после подтверждения ремонт становится полноценной записью в системе

### Вопрос 18

Что именно в MVP должно считаться подозрительным ремонтом: повторный ремонт, превышение нормо-часов, завышенная сумма, дубли строк, расхождение итогов или всё перечисленное?

- по каждой проверке должен формироваться статус и причина
- в карточке ремонта и в аналитике должны отображаться сработавшие флаги

### Вопрос 19

Как определять повторный ремонт: если по одной и той же машине повторяется такая же работа в течение какого периода?

### Ответ 19

Повторный ремонт считается подозрительным, если по одной и той же машине повторяется такая же работа в течение 30 дней.

### Вопрос 20

Какой порог считать подозрительным по нормо-часам: если фактические нормо-часы превышают справочные на сколько процентов?

### Ответ 20

Подозрительным по нормо-часам считается превышение справочного значения более чем на 10%.

Следствие для реализации:

- при наличии сопоставления со справочником нужно считать отклонение факта от нормы
- если отклонение больше 10%, ремонт помечается как подозрительный по трудозатратам
- правило должно быть вынесено в настраиваемый параметр на будущее

### Вопрос 21

Какой порог считать подозрительным по сумме ремонта: если сумма выше ожидаемой, то на сколько процентов?

### Ответ 21

Подозрительной по сумме считается стоимость ремонта, если она превышает ожидаемое значение более чем на 10%.

Следствие для реализации:

- потребуется расчёт ожидаемой суммы как базы для сравнения
- порог отклонения по сумме для MVP зафиксирован как 10%
- правило также нужно сделать настраиваемым в будущем

### Вопрос 22

От какой базы считать ожидаемую сумму ремонта: от средней по таким же работам, от нормо-часов и ставки, или по другому правилу?

### Ответ 22

Решение для MVP выбирает исполнитель.

Рабочее правило:

- если по такой же работе уже есть история, ожидаемую сумму считать по медиане или средней стоимости аналогичных ремонтов
- в первую очередь сравнивать по совпадению работы
- дополнительно учитывать сервис и тип техники, если данных уже достаточно
- если истории недостаточно, использовать fallback-расчёт от нормо-часов и базовой ставки работ

Причина выбора:

- исторические данные дают более реалистичную базу, чем чисто нормативные часы
- fallback через нормо-часы позволяет работать даже на старте, когда накопленной статистики ещё мало
- такая схема подходит для постепенного улучшения аналитики без переделки архитектуры

### Вопрос 23

Где в MVP хранить PDF-файлы заказ-нарядов: локально на сервере или сразу в S3-совместимом хранилище?

### Ответ 23

Решение для MVP выбирает исполнитель.

Принятое решение:

- в production PDF-файлы хранить в S3-совместимом хранилище
- в локальной разработке и тестовой среде допускать хранение на файловой системе

Причина выбора:

- S3-совместимое хранилище лучше подходит для надёжного хранения документов
- упрощает резервирование, масштабирование и перенос между серверами
- не привязывает проект к локальному диску одного сервера
- локальный режим разработки сохраняет простоту запуска MVP

- повышает ценность карточки ремонта и карточки техники

### Вопрос 26

### Ответ 26

С учётом уже предоставленных материалов для MVP лучше загружать исторические данные по прошлым периодам, а не начинать только с новых ремонтов после запуска.

- уже есть справочник нормо-часов Dong Feng

### Вопрос 112

Нужно ли после этого перейти к следующему этапу и на основе всех зафиксированных ответов подготовить для проекта структурированный план реализации, архитектуру модулей и список ближайших шагов?

### Ответ 112

Да, после завершения сбора требований нужно перейти к следующему этапу и подготовить:

- структурированный план реализации
- архитектуру модулей
- список ближайших шагов
