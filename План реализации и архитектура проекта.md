## Статус

Критичных блокеров для старта разработки нет.

Актуальный эталон пользовательского сценария сотрудника и обязательный pre-deploy чек-лист вынесены в отдельный документ:

## Цель MVP

Сделать внутреннюю веб-систему учёта ремонтов автопарка, в которой:

- техника загружается из реестров
- документы по ремонтам загружаются вручную из PDF, фото и сканов
- система автоматически распознаёт документы
- пользователь проверяет и дополняет распознанные данные
- ремонт проходит двухступенчатое подтверждение
- по машине накапливается полная история
- система строит аналитику, визуальные дашборды и контроль качества данных

Ключевой пользовательский сценарий MVP:

- сотрудник загружает заказ-наряд
- система распознаёт и валидирует данные
- сотрудник проверяет и корректирует результат
- система формирует предупреждения по несоответствиям
- сотрудник получает итоговый отчёт по конкретному заказ-наряду

## Выбранная архитектура

### Технологический стек

- Frontend: `React + TypeScript + Vite`
- UI: `MUI` с акцентом на внутренние плотные рабочие интерфейсы
- Backend: `FastAPI`
- ORM: `SQLAlchemy`
- Миграции: `Alembic`
- База данных: `PostgreSQL`
- Аутентификация: `JWT + bcrypt`
- Файлы: `S3-совместимое хранилище` в production, локальное хранилище в dev
- Деплой: `Docker Compose` на Linux-сервере

### Логическая схема

- Frontend
- Backend API
- PostgreSQL
- S3-совместимое файловое хранилище
- OCR и пайплайн распознавания документов
- очередь фоновой обработки документов

## Основные модули MVP

### 1. Аутентификация и пользователи

- вход по логину и паролю
- восстановление пароля через e-mail
- ручное создание пользователей через админку
- роли:
  - `администратор`
  - `сотрудник`
- обязательный `e-mail` у пользователя

### 2. Техника

- единая модель техники с типом:
  - `грузовик`
  - `прицеп`
- импорт техники из:
  - `Грузовики Парк.xls`
  - `Прицепы Парк.xls`
- карточка техники
- статусы техники:
  - `в эксплуатации`
  - `в ремонте`
  - `ожидает ремонта`
  - `неактивна`
  - `списана`

### 3. История техники

Для каждой единицы техники хранить:

- историю ремонтов
- историю документов
- историю привязки к сотрудникам
- историю привязки к механикам
- историю связок грузовик ↔ прицеп
- пробег на момент ремонта

### 4. Документы и OCR

- загрузка `PDF`
- загрузка фото и сканов
- мобильная загрузка с телефона
- объединение нескольких фото в один документ
- OCR обязателен
- язык документов на MVP: `русский`
- целевой порог качества OCR: `90%` по ключевым полям
- допустимое время обработки документа: `до 10 секунд`

### 5. Экран проверки документа

- split-view:
  - оригинал документа
  - распознанные поля
- подсветка места в документе, откуда взято поле
- уверенность OCR по документу и по полям
- ручное исправление и дополнение полей
- обязательные поля для подтверждения:
  - машина
  - дата
  - сервис
  - номер заказ-наряда
  - итоговая сумма
  - пробег

### 6. Ремонты

- ремонт всегда создаётся на основании документа
- источником может быть:
  - заказ-наряд
  - фото/скан распознаваемого документа
- один ремонт может иметь несколько файлов
- поддержка версий документа
- при новой версии:
  - повторный OCR
  - пересчёт проверок
  - сохранение прошлой подтверждённой версии

### 7. Жизненный цикл ремонта

- `черновик`
- `на проверке`
- `подозрительный`
- `подтвержден`
- `ошибка распознавания`
- `архивный`

Подтверждение двухступенчатое:

- сотрудник может подтвердить ремонт по своим данным
- финальное подтверждение делает администратор
- до финального подтверждения ремонт учитывается в аналитике как предварительный

### 8. Документы и статусы документа

- `загружен`
- `распознан`
- `частично распознан`
- `требует проверки`
- `подтвержден`
- `ошибка распознавания`
- `архивирован`

Нужна отдельная очередь документов на ручную проверку.

Приоритет в очереди:

- сначала подозрительные случаи

### 9. Справочники

- сервисы
- нормативные работы Dong Feng
- материалы и позиции из истории и документов

Особенности:

- сервис может создать и сотрудник по своим данным
- сервис, созданный сотрудником, подтверждает администратор
- до подтверждения сервис участвует в ремонтах и аналитике как предварительный
- новые работы и материалы вне справочников также подтверждает администратор
- до подтверждения они тоже допускаются как предварительные

### 10. Импорт исторических данных

- пакетный импорт через админку
- импорт техники
- импорт исторических ремонтов и связанных данных из доступных Excel
- конфликты импорта не перезаписываются автоматически
- конфликты отправляются на ручную проверку
- сотрудник видит конфликты только по своей технике
- администратор видит все конфликты

### 11. Нормализация и дубли

- администратор объединяет дубли сервисов, работ и материалов
- все связи автоматически переносятся на итоговую запись
- сохраняется история объединения
- объединение можно откатить администратором

### 12. Аудит и журнал действий

- отдельный журнал действий в интерфейсе
- фильтры:
  - пользователь
  - тип действия
  - сущность
  - период
- фиксируются:
  - создание
  - изменение
  - подтверждение
  - архивирование
  - восстановление
  - импорт
  - объединение дублей
  - откат объединения

### 13. Аналитика

Обязательны визуальные дашборды.

Периоды:

- месяц
- квартал
- год
- произвольный диапазон дат
- текущий момент

Приоритет аналитики:

- всё, что связано с заказ-нарядом

Ключевые аналитические блоки:

- расходы по машинам
- расходы по сервисам
- подозрительные ремонты
- загрузка сотрудников
- карточки KPI
- графики
- таблицы

### 14. Дашборд качества данных

Отдельный модуль качества данных:

- качество OCR
- очередь проверки
- неподтверждённые сервисы
- неподтверждённые работы
- неподтверждённые материалы
- конфликты импорта

Доступ:

- администратор видит всё
- сотрудник видит только свою зону ответственности

### 15. Поиск

Глобальный поиск по:

- VIN
- госномеру
- номеру заказ-наряда
- сервису
- артикулу
- названию работы

Приоритет выдачи:

- заказ-наряды
- ремонты
- машины

### 16. Экспорт

Форматы:

- `Excel`
- `PDF`

Объекты экспорта:

- списки
- аналитика
- карточка машины
- карточка ремонта

### 17. Архив

- удаление не используется
- только архивирование
- после ликвидации автомобиля техника и её история уходят в архив
- архив остаётся доступным для:
  - поиска
  - просмотра
  - экспорта

## Основные бизнес-правила подозрительности

Подозрительный ремонт:

- повторный ремонт по одной машине и такой же работе в течение `30 дней`
- превышение нормо-часов более чем на `10%`
- превышение ожидаемой суммы более чем на `10%`
- дубли строк
- расхождение сумм
- аномалия пробега

Дополнительно:

- любое расхождение сумм считать подозрительным
- допустимое отклонение по суммам: `0`
- если новый пробег меньше предыдущего по той же машине, это подозрительный случай
